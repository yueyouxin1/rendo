# V1 发布门槛清单与交付成功定义

## 结论

`rendo cli`、模板体系和 Rendo runtime 必须一起进入可用状态，才算 `v1`。

单独发布 CLI、不带远程 runtime、不带狗粮验证，不算正式 `v1`。

## 一、V1 发布门槛清单

### A. 模板体系

- [ ] `core -> base -> derived` 主干冻结一版
- [ ] 五类一等模板类型语义冻结一版
- [ ] 五类 `core` 模板可稳定初始化
- [ ] 五类官方 `base` 模板可稳定被搜索和检查
- [ ] 至少一个官方 derived SaaS starter 可承载 Rendo 自身 runtime

### B. CLI

- [ ] `rendo init <kind>` 正式可用
- [ ] `rendo create` 正式可用
- [ ] `rendo search / inspect / pull` 支持远程 registry
- [ ] `rendo add / upgrade` 至少对首批官方资产正式可用
- [ ] CLI 对下载制品做 digest 校验
- [ ] CLI 有明确版本策略和兼容矩阵

### C. Headless Rendo Runtime

- [ ] 有远程 registry API
- [ ] 有 PostgreSQL 持久化元数据
- [ ] 有对象存储制品仓
- [ ] 有 publish / search / inspect / download 闭环
- [ ] 有模板版本、渠道和弃用策略
- [ ] 有最小鉴权、审计和监控

### D. 狗粮验证

- [ ] Rendo 自己通过这套 runtime 发布模板
- [ ] Rendo 自己通过正式 CLI 消费模板
- [ ] derived SaaS starter 能运行 Rendo 自身核心后端逻辑
- [ ] 核心链路至少经过一次真实升级验证

### E. 运维与发布

- [ ] 有本地开发方案
- [ ] 有 Docker-first 部署方案
- [ ] 有迁移、备份、恢复说明
- [ ] 有失败回滚或安全中止策略
- [ ] 有最小文档交付包

## 二、交付成功的定义

以下条件同时满足，才算 `v1` 交付成功：

1. 一个新用户或强 Agent 不需要当前源码仓，也能通过正式 CLI 从远程获取官方模板。
2. 该用户可以通过 `rendo init <kind>` 初始化 core 模板，并通过 `rendo create` 创建 starter 项目。
3. 该用户可以对官方 capability / provider / surface 资产执行 `search / inspect / pull / add / upgrade` 的核心流程。
4. Rendo runtime 自己就是通过这套模板体系和 CLI 机制被支撑和验证的，而不是旁路实现。
5. 核心契约在发布后不会因为“模板体系无法支撑 Rendo 自身”而立即大改。

## 三、不满足就不能叫 V1 的情况

出现以下任一情况，都不应称为正式 `v1`：

- 只有本地 registry，没有远程 runtime
- CLI 只能在当前仓库里工作
- 模板能创建 demo，但不能承载 Rendo 自身核心后端
- runtime 还没有持久化元数据和制品存储
- 升级、兼容、制品完整性没有正式规则

## 四、V1 之后再做的东西

以下可以放到 `v1` 之后：

- SaaS 前端控制台
- 模板 marketplace UI
- 企业管理面
- 更复杂的商业化流程
- 更完整的多端产品层
