# 从模板反向长平台的适用边界与前置契约说明

> 术语说明：本文件中若出现历史表述 `Core Starter`，在当前架构下应优先按“对应模板类型的 `core` 模板”理解；只有明确讨论 starter 类型时，才特指 `starter-core-template`。

- 文档版本：v1.0
- 日期：2026-04-04
- 文档性质：原则修正文档
- 目标：明确“从模板反向长出 Rendo 平台”这条思路哪些部分仍然成立，哪些部分必须修正，并说明为什么最小平台契约必须先于模板定义
- 核心结论：**“从模板反向长平台”仍然成立，但只适用于平台功能面；平台的最小核心契约层必须先定义，否则最佳实践模板会在后续大面积返工。**

---

## 1. 问题定义

此前有一个重要判断：

- **Rendo 平台应从最佳实践模板中反向长出来**

这个判断在当时成立，是为了避免重新掉回：

- 先做大平台
- 再往里填内容

的旧路径。

但随着后续继续梳理，我们已经发现一个新的现实问题：

**要做出真正可用的最佳实践模板，本身就需要先知道 Rendo 平台最小的能力边界、接口、数据契约和扩展机制。**

这说明，原来的说法虽然方向对，但表达得不够精确。

---

## 2. 原说法为什么不再够用

如果把“从模板反向长平台”理解成：

- 什么都不先定义
- 直接开始做模板
- 平台后面自然长出来

这条路现在已经不成立。

原因很简单：

## 2.1 模板不是孤立文件，而是平台资产

新的 Rendo 模板不是一堆源码而已，它还需要：

- metadata
- capability declarations
- pack 依赖声明
- runtime mode
- 初始化方式
- 文档和 agent instructions

这些都天然依赖某种平台语义。

## 2.2 模板需要知道“平台怎么理解它”

例如：

- starter manifest 长什么样
- pack manifest 长什么样
- source / managed / hybrid 怎么定义
- API key 怎么接
- CLI 如何初始化
- pack 安装如何修改工程

如果这些不先定义，模板做出来也会不稳定。

## 2.3 没有契约先行，后续统一会非常痛苦

如果直接做模板，后面会很容易发生：

- 每个模板 metadata 不一致
- 每个模板 pack 引用方式不一致
- 每个模板 docker 和 env 结构不一致
- Agent instructions 风格不一致

到时候平台再来统一，就会付出很高返工成本。

---

## 3. 正确的修正说法

因此，之前的判断要修正成：

**平台的产品功能面可以从模板反向长出来，但平台的最小核心契约层必须先定义。**

这句话可以拆成两部分理解。

## 3.1 可以反向长出的部分：平台功能面

这些确实应该后长出来：

- 模板搜索
- 模板预览
- 下载 / 拉取体验
- 模板版本展示
- 模板升级路径
- 企业模板库
- Pack Registry UI
- 作者后台
- 授权和分发产品面

这些都应该从：

- 模板的真实使用
- Agent 的真实接入
- 用户的真实分发需求

中自然长出来。

## 3.2 必须先定义的部分：平台最小契约层

这些不能等模板做完再想：

- `Core Starter` 结构契约
- `Starter Template` 结构契约
- `Capability Template` manifest 契约
- runtime mode 语义
- CLI / create / add / inspect 协议
- metadata / docs / agent instructions 规范
- config / env 命名约定
- provider / adapter 基础抽象

这些东西不是平台功能，而是：

- **模板得以稳定存在的基础规则**

---

## 4. 推荐的正确顺序

基于这个修正，Rendo 的正确顺序应当是：

## 第一步：定义最小平台契约

只定义最少但必须的规则：

- starter contract
- manifest contract
- pack contract
- runtime mode
- CLI 行为
- 默认目录规范
- metadata 规范

注意：

- 这一步不是做完整平台
- 这一步只是定规则

## 第二步：基于契约做出第一个高质量模板

这时模板就不是自由发挥，而是建立在可持续扩展的标准之上。

## 第三步：用模板验证契约是否合理

让真实业务和强 Agent 去使用 starter，暴露：

- 契约不足
- 不自然之处
- 冗余设计

然后修正契约。

## 第四步：从模板反向长出平台功能面

当模板与契约都稳定后，再长出：

- 搜索
- 预览
- 授权
- 升级
- 管理

这些平台功能。

---

## 5. 为什么这种顺序更合理

## 5.1 它避免平台先行

不会回到“先造平台再找内容”的老路。

## 5.2 它避免模板野蛮生长

模板不会在没有标准的情况下长成一堆不可统一的资产。

## 5.3 它让契约接受真实模板检验

契约不是闭门设计，而是马上被第一个 starter 验证。

## 5.4 它让平台功能面真正从需求中长出来

平台不是先想象出来，而是从真实 starter 需求里长出来。

---

## 6. 对当前工作的直接影响

当前不应再把任务理解成：

- 直接做 starter

而应理解成：

- 先补齐最小平台契约
- 再做 starter

但这里的“平台契约”必须控制得非常轻，不允许再次滑回：

- 先做平台产品

否则又会偏离方向。

## 当前最小前置契约至少包括

1. `Core Template` 目录与约定
2. `Starter Template` 分类和继承关系
3. `rendo.template.json` 的字段与语义
4. `pack manifest` 的字段与语义
5. `runtime mode` 定义
6. `rendo create / add / inspect` 的最小协议
7. Agent 指令文件规范

---

## 7. 最终结论

“从模板反向长平台”这条思路并没有失效，  
但必须做一个关键修正：

- **平台功能面可以后长**
- **平台核心契约层必须先定义**

一句话结论：

**不能先做完整平台，但也不能在没有最小平台契约的情况下直接做模板。**
