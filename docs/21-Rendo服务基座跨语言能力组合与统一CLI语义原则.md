# Rendo 服务基座跨语言能力组合与统一 CLI 语义原则

- 文档版本：v2.0
- 日期：2026-04-06
- 文档性质：架构原则文档
- 目标：说明 Rendo 在多语言生态下应该如何组织服务基座和能力模板，并保持统一认知
- 核心结论：**Rendo 不应被实现为“每种语言各自一套平行体系”，而应被实现为“统一契约 + 统一 Agent 接口面 + 统一 CLI 语义 + 多语言能力组合系统”。**

---

## 1. 问题定义

随着新定位继续收敛，一个关键问题越来越明确：

- Rendo 已经不再局限于 Web、`Next.js` 或单一语言生态
- 很多真实的面向智能体服务天然就是多语言组合

例如：

- 某些部分更适合用 `TypeScript`
- 某些部分更适合用 `Python`
- 某些端需要 Web 壳
- 某些能力是 headless 的

因此，Rendo 面临的真实问题不是：

- 如何让一种语言统治全部 starter

而是：

- **如何在多语言、多形态、多能力组合的情况下，依然保持清晰、一致、可被强 Agent 理解。**

---

## 2. 核心判断

### 2.1 不该做什么

Rendo 不应做成：

- 每种语言一套独立 starter 世界
- 每种语言一套独立 CLI 语义
- 每种语言一套独立能力模板生态
- 每种语言一套独立平台规则

### 2.2 应该做什么

Rendo 应该把以下内容统一下来：

- 契约
- CLI 语义
- 模板描述方式
- 能力模板扩展方式
- runtime mode 语义
- Agent 接口面

而把以下内容交给具体模板实现：

- 具体语言
- 具体框架
- 具体目录落位
- 具体实现方式

一句话：

**统一的是规则，不是实现。**

---

## 3. 什么叫“跨语言能力组合”

更合理的理解是：

- 一个 starter 服务基座可以由多种语言与多种执行单元组成
- 一个能力模板也可以作用于不同语言实现
- 只要它们遵守统一契约和统一 CLI 语义，就应被视为同一体系的一部分

所以，“跨语言能力组合”本质上是在回答：

- **不同语言如何在一个服务基座体系里自然共存**

---

## 4. 为什么统一 CLI 语义是关键

强 Agent 最不适合面对的情况是：

- 每种语言一套命令
- 每种 starter 一套行为
- 每种能力模板一套安装方式

它最适合面对的是：

- 相同命令
- 相同语义
- 相同可见性
- 相同失败方式

这意味着：

- `rendo init`
- `rendo create`
- `rendo inspect`
- `rendo add`
- `rendo pull`
- `rendo upgrade`
- `rendo doctor`

这些命令的语义必须跨语言保持一致。

---

## 5. 哪些东西必须统一

### 5.1 Starter 身份语义

必须统一：

- 什么是 `starter-core-template`
- 什么是 `Starter Template`
- 一个 starter 如何声明自己的类型、能力和默认依赖
- 一个 starter 如何声明 `AGENTS.md / .agents / interfaces/*`

### 5.2 Capability Template 语义

必须统一：

- 什么是能力模板
- 能力模板如何声明类型
- 能力模板如何声明运行模式
- 能力模板如何声明安装影响面

### 5.3 Runtime mode 语义

必须统一：

- `source`
- `managed`
- `hybrid`

的定义和边界。

### 5.4 CLI 语义

必须统一：

- `init` 是初始化底座
- `create` 是创建 starter 服务基座
- `inspect` 是查看 starter 或能力模板说明
- `add` 是安装能力
- `doctor` 是诊断环境和依赖

### 5.5 配置与依赖表达方式

必须统一：

- 配置入口的概念
- provider 的声明方式
- 必要环境变量的表达方式
- install / upgrade / remove 的基本逻辑
- Agent 接口面的扩展方式

---

## 6. 哪些东西不应该写死

为了避免把实现思维限制死，以下部分不应在原则层写死：

1. 固定目录名称
2. 固定一门语言做 UI
3. 固定一门语言做服务
4. 固定某种“app/service”分法
5. 固定某种单仓目录长相

原因很简单：

- 这些都属于实现策略
- 会随着 starter 场景和语言生态变化
- 过早写死会限制工程选择

---

## 7. 怎样做才算“做对”

如果 Rendo 要把跨语言能力组合做对，至少应满足以下条件：

### 条件 1：Agent 看到不同语言 starter 时，不需要重新学习一套世界

它应该仍然能通过：

- manifest
- CLI
- docs
- install plan

快速理解这个 starter。

### 条件 2：不同语言实现仍然能被纳入同一套 starter 与能力模板体系

无论某个能力是：

- `TypeScript`
- `Python`
- 其他生态

都必须可以被 starter 与能力模板体系统一描述。

### 条件 3：组合能力不应破坏 starter 的可理解性

即使一个 starter 是多语言的，也必须：

- 结构清晰
- 依赖明确
- 安装路径明确
- Agent 接口面清晰

### 条件 4：CLI 必须优先于底层实现差异

不同语言实现可以不同，  
但用户和 Agent 的操作方式不应该跟着语言大幅变化。

### 条件 5：契约必须先于实现

先定义：

- starter contract
- 能力模板 contract
- runtime mode
- CLI semantics
- Agent 接口 contract

再去做多语言 starter。

---

## 8. 对 starter core 的影响

这个原则进一步说明：

`starter-core-template` 的“语言无关”应理解为：

- 它定义统一工程语言和扩展规则

而不是：

- 它自己要实现所有语言版本

所以：

- `starter-core-template` 应足够抽象
- `Starter Template` 才去绑定具体实现形态

---

## 9. 最终结论

Rendo 在跨语言场景下最重要的不是：

- 支持多少语言

而是：

- 是否能在多语言现实下依然保持统一认知、统一 CLI、统一 manifest、统一 Agent 接口面和统一扩展机制

一句话结论：

**Rendo 真正要做的不是“多语言 starter 集合”，而是“在统一契约、统一 Agent 接口面和统一 CLI 语义之上的多语言能力组合系统”。**
