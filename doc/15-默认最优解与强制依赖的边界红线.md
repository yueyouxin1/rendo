# 默认最优解与强制依赖的边界红线

- 文档版本：v1.0
- 日期：2026-04-04
- 文档性质：边界红线文档
- 目标：明确 Rendo 在“默认最优解”与“强制依赖”之间的边界，防止 starter、pack 和平台设计逐步滑向半封闭平台模式
- 核心结论：**Rendo 可以提供默认最优解，但不能把默认最优解做成事实上的强制依赖。`Core Starter` 必须零强依赖，`Domain Starter` 可以 `Rendo-optimized`，官方 pack 可以默认最好用，但必须为退出和替换保留明确路径。**

---

## 1. 为什么必须写这份文档

Rendo 当前的设计同时追求几件事：

- starter 默认可运行
- 小白用户尽量少配置
- 强 Agent 能自然扩展能力
- 官方能力和统一 API Key 尽量形成默认最优解

这些目标本身没有问题。  
但如果没有边界，很容易自然滑向：

- 看起来开放
- 实际上强依赖平台

这会带来两个后果：

1. 对用户和强 Agent 来说，不再自然
2. Rendo 会重新回到“平台锁定优先”的旧路径

因此必须提前定义红线。

---

## 2. 两个概念必须严格区分

## 2.1 什么叫默认最优解

默认最优解是指：

- starter 默认接入 Rendo 官方能力
- 默认路径最省事
- 默认路径最稳定
- 默认路径最容易升级
- 用户可以迁移，但通常没有必要第一时间迁移

核心特征是：

- **可替换**
- **可退出**
- **可继续开发**

只是：

- 默认方案明显更划算

## 2.2 什么叫强制依赖

强制依赖是指：

- 不接 Rendo 就根本跑不起来
- 不接官方服务就丢掉核心功能
- 没有 source / local fallback
- pack / starter 的关键语义只在平台里成立

核心特征是：

- **理论开放**
- **实践上离不开**

这就是 Rendo 必须避免的状态。

---

## 3. 哪些情况说明已经开始滑向强制依赖

以下任一情况出现，都说明设计已经开始越界。

## 3.1 `Core Starter` 启动时必须要求 `Rendo API Key`

如果没有 `Rendo API Key`，`Core Starter` 就无法启动或无法完成基本功能，  
那 `Core Starter` 就已经不是基础底座，而是平台客户端。

## 3.2 `Core Starter` 的关键目录和语义只依赖平台控制面

如果：

- starter 的真实状态只在云端
- manifest 只是平台控制面的影子
- 本地工程无法独立表达关键配置

那 starter 已经失去源码路线意义。

## 3.3 官方 managed pack 成为关键功能唯一实现

如果某些核心能力：

- 不存在 source / hybrid 替代
- 没有官方平台就无法运行

那这些能力就构成了事实上的平台锁定。

## 3.4 默认能力的退出路径不存在或过于模糊

如果对用户来说：

- 理论上可以换
- 但实际上不知道怎么换
- Agent 也不知道怎么 eject

那这同样构成强依赖。

## 3.5 模板元数据和 CLI 只服务于 Rendo 平台，不服务于工程本身

如果 manifest、CLI、pack 规范只是为了平台方便，而不再服务于：

- 本地工程
- Agent 阅读
- Git 管理

那 starter 就已经从工程资产退化成平台资产。

---

## 4. 核心红线

这里给出必须遵守的红线。

## 红线 1：`Core Starter` 必须零强依赖

`Core Starter` 必须满足：

- 没有 `Rendo API Key` 也能工作
- 没有官方 managed service 也能工作
- 不依赖平台 UI 和平台控制面

它只应：

- 理解 Rendo 协议
- 兼容 Rendo 能力
- 预留 Rendo 接入点

但不能：

- 把自己变成 Rendo Cloud 客户端

## 红线 2：`Domain Starter` 可以 `Rendo-optimized`，但不应普遍 `Rendo-required`

`Domain Starter` 可以默认：

- 接官方 pack
- 接官方 provider
- 接官方默认能力

但不应大量出现：

- 一旦脱离 Rendo 官方能力就核心功能全部失效

如果某些 starter 确实必须强依赖 Rendo，也必须明确标记：

- `rendo-required`

且不能作为默认主推 starter。

## 红线 3：关键能力必须尽量存在 source 或 hybrid 路径

不是每个能力都必须完全开放源码。  
但对高频关键能力，应尽量保证：

- `source`
- 或 `hybrid`

至少存在一种。

这样：

- 高级用户和强 Agent 才有保底退出路径

## 红线 4：Pack 不能破坏工程主权

pack 安装后：

- 工程仍应主要是本地可读写工程
- pack 的 effect 必须进入 Git / diff / review
- Agent 必须能在本地继续理解和改造它

pack 不能把核心逻辑都偷偷搬进云端。

## 红线 5：平台价值必须来自“更好”，而不是“离不开”

Rendo 官方能力可以：

- 更快
- 更稳
- 更省心
- 更方便升级

但不应依赖：

- 让用户无法迁移
- 让 Agent 无法替换

---

## 5. `Core Starter / Domain Starter / Pack` 的正确边界

## 5.1 `Core Starter`

允许：

- 兼容 Rendo
- 理解 Rendo
- 预留 Rendo 能力入口

不允许：

- 默认强依赖 Rendo 平台

## 5.2 `Domain Starter`

允许：

- 默认接官方能力
- 默认推荐官方 provider
- 默认以 `hybrid` 方式运行

不应：

- 默认全部做成 `managed-only`

## 5.3 `Capability Pack`

允许：

- 官方 pack 提供最佳体验
- 统一 API Key 降低配置成本
- 小白用户优先使用 managed pack

但必须：

- 清楚声明 `source / managed / hybrid`
- 明确 install plan
- 明确依赖等级

---

## 6. 允许存在的“高迁移成本”应该是什么

Rendo 并不是不能有迁移成本。  
只是迁移成本应来自：

- 默认体验更好
- 升级更顺
- 运维更省心
- 官方能力更稳定
- 官方模板和官方 pack 配合更自然

而不是来自：

- 藏起来的依赖
- 不可见的云端状态
- 无法替换的核心接口

一句话：

**应有健康迁移成本，不应有事实锁定。**

---

## 7. 实践判断标准

在设计 starter 或 pack 时，可以用以下问题做检查：

1. 没有 Rendo API Key，这个 starter 还能不能最小运行？
2. 如果去掉官方 pack，Agent 是否仍然有替代路径？
3. 核心逻辑是否仍然存在于本地可读写工程中？
4. manifest 和 CLI 是否仍然首先服务于工程，而不是只服务于平台？
5. 用户继续订阅是因为“更好”，还是因为“离不开”？

只要其中多数答案偏向后者，就说明设计已经越界。

---

## 8. 最终结论

Rendo 的价值可以建立在：

- 默认最优解
- 统一 API Key
- 官方 pack 最省事
- 官方云能力最稳

这些都没有问题。  
真正必须防止的是：

- 这些默认优势逐渐演化为事实上的强制依赖

一句话结论：

**`Core Starter` 必须零强依赖，`Domain Starter` 只可优化不可全面绑死，官方 pack 可以最好用，但必须永远为用户和强 Agent 保留退出与替换路径。**
