# Rendo 跨语言能力组合交付标准与场景定义

- 文档版本：v1.0
- 日期：2026-04-04
- 文档性质：交付标准文档
- 目标：定义 Rendo 跨语言能力组合真正“做成”的标准，明确从用户视角和内部实现视角看，成功是什么样子
- 核心结论：**Rendo 的跨语言能力组合并不是“支持多几种语言”就算成功，而是让用户和强 Agent 能在统一 CLI、统一契约和统一 starter / pack 语义下，自然组合多语言能力而不产生认知断裂。**

---

## 1. 为什么需要交付标准

“支持跨语言”很容易停留在口号层。

例如：

- 一个 starter 里既有 Node 又有 Python
- pack 里支持两套语言实现
- CLI 也勉强能跑

这些都不等于真正交付成功。

真正需要回答的是：

1. 用户能否自然使用
2. 强 Agent 能否自然理解
3. starter 是否仍然清晰可维护
4. 平台和 CLI 是否能统一描述这些组合

所以，这份文档的目标不是再解释原则，而是定义：

- **什么时候算做成**

---

## 2. 成功定义

Rendo 跨语言能力组合“做成了”，至少要同时满足以下四层成功条件。

## 2.1 用户心智成功

用户不需要理解：

- 为什么这里是 Node
- 为什么那里是 Python
- 为什么这个能力在另一个 runtime 中

用户只需要理解：

- 我要的 starter 是什么
- 我加了什么能力包
- 它们都已经接好了
- Agent 可以继续往下做

如果用户仍然被迫从“业务意图”切换到“语言和运行时意图”，说明没有做成。

## 2.2 Agent 心智成功

强 Agent 在面对多语言 starter 时，不应像面对多个独立项目。  
它应仍然能通过统一入口理解：

- starter 是什么
- pack 是什么
- 哪个模块承担什么职责
- 怎样初始化、扩展、诊断、升级

如果 Agent 需要分别学习：

- Node 一套世界
- Python 一套世界
- 各种语言各一套命令和 manifest

那说明没有做成。

## 2.3 工程结构成功

多语言 starter 虽然复杂，但必须仍然：

- 结构清晰
- 依赖关系清晰
- 运行边界清晰
- 安装和升级路径清晰

如果语言一多，starter 就迅速失去可读性和可维护性，也说明没有做成。

## 2.4 平台与 CLI 成功

平台、CLI、manifest、pack 必须仍然用同一套语言描述这些能力。  
如果不同语言 starter 需要不同的：

- CLI 语义
- 模板描述方式
- pack 安装逻辑

也说明没有做成。

---

## 3. 最低交付标准

下面是更具象的最低交付标准。

## 3.1 统一入口标准

无论 starter 最终包含几种语言，用户和 Agent 都应通过同一套入口工作：

- `rendo init`
- `rendo create`
- `rendo inspect`
- `rendo add`
- `rendo pull`
- `rendo upgrade`
- `rendo doctor`

只要某种语言能力要求额外另一套入口，就不算达标。

## 3.2 统一 manifest 标准

无论能力由什么语言实现，starter 与 pack 都必须能被：

- `rendo.template.json`
- `pack manifest`

统一描述。

必须明确：

- 能力是什么
- 依赖什么
- 运行模式是什么
- 安装会影响什么

## 3.3 统一运行模式标准

无论底层语言如何，仍然必须清楚区分：

- `source`
- `managed`
- `hybrid`

如果跨语言后这些语义变模糊，说明没有做成。

## 3.4 统一诊断标准

`rendo doctor` 至少要能告诉用户和 Agent：

- 哪些语言 runtime 需要什么环境
- 哪些模块已经配置好
- 哪些 pack 已安装
- 哪些服务未启动
- 哪些 provider 缺失

如果跨语言一多就无法统一诊断，也不算达标。

## 4. 从用户视角看，成功是什么样子

下面用具象场景定义“看起来像成功”的体验。

## 场景 A：SaaS + Python AI 服务

### 用户目标

用户想做一个 AI SaaS：

- Web 前端和后台管理用 Web 技术
- AI 能力和数据处理用 Python

### 用户行为

用户或强 Agent 执行：

```bash
rendo create ai-saas-starter
rendo add python-ai-service-pack
```

### 用户感知

用户看到的是：

- starter 已创建
- Docker 工作区可启动
- Web 端可访问
- AI 服务也已接好
- 不需要自己理解“为什么这部分是 Python”

### 用户不应该被迫做的事

用户不应被迫：

- 先自己规划多语言目录
- 自己决定两个语言怎么接
- 自己配置多套完全不同的启动流程

如果这些都还需要手工做，就说明没有真正交付成功。

## 场景 B：Web starter 后续加小程序能力

### 用户目标

已有一个 Web starter，后续需要加一个小程序壳。

### 用户行为

用户或 Agent 可以通过：

- 选择支持 miniapp 的 domain starter
- 或安装一个 miniapp 相关能力包

### 用户感知

用户不需要理解：

- 前端壳为什么不同
- 哪个运行时承担了什么

用户只会感知到：

- 项目现在多了一种端
- 核心业务能力还是同一个

如果用户必须先理解复杂的多端工程细节，说明没做好。

## 场景 C：Headless 项目加 Web 管理面板

### 用户目标

一个原本 headless 的 automation / workflow 工程，后续需要加一个轻量 Web 管理面板。

### 成功体验

用户或 Agent 可以把“Web 管理面板”当成一种可组合能力，而不是必须重做整个项目结构。

这说明跨语言和跨形态能力组合已经自然成立。

---

## 5. 从强 Agent 视角看，成功是什么样子

对强 Agent 而言，成功的核心不是“支持很多语言”，而是：

- **项目仍然像一个统一工程，而不是一堆碎工程。**

### Agent 成功体验应包括

1. 能通过 manifest 识别每个模块承担的职责
2. 能通过 CLI 做初始化、安装、升级、诊断
3. 不需要猜测“这个目录到底是什么语言和用途”
4. 能明确知道 pack 会影响哪些模块
5. 能在多语言项目中仍然维持清晰修改边界

### Agent 不应遇到的情况

1. 每种语言一套完全不同命令
2. 没有清晰模块描述
3. pack 修改了很多地方但 install plan 不透明
4. 启动方式和 env 规则高度分裂

只要这些问题存在，说明强 Agent 体验还没成立。

---

## 6. 从 Rendo 内部实现看，成功是什么样子

Rendo 内部“做成了”的标志，不是支持了多少语言，而是：

## 6.1 契约仍然统一

无论底层语言是什么，平台和 CLI 看到的 starter / pack 都仍然是：

- 同一套 manifest 语义
- 同一套 runtime mode 语义
- 同一套 install plan 语义

## 6.2 CLI 仍然是第一入口

不是：

- Node 项目用一套 CLI
- Python 项目用另一套 CLI

而是：

- 同一 CLI
- 根据 manifest 和安装计划做不同语言实现

## 6.3 Starter 仍然能逐步升级

跨语言组合不应让 starter 失去：

- 升级能力
- 安装能力
- 可维护性

## 6.4 平台仍然能统一治理

即使 starter 内部是多语言组合，平台仍然应能：

- 展示它是什么
- 展示支持什么
- 展示用了哪些 pack
- 展示推荐升级路径

---

## 7. 用户最终得到什么

如果这项能力交付成功，用户真正得到的不是：

- “多语言支持”这个抽象能力

而是以下实际收益：

### 1. 更自然的 starter 组合能力

用户不用为了技术栈妥协业务场景。

### 2. 更少的从零搭桥成本

语言与能力组合不再需要每次自己拼。

### 3. 更低的强 Agent 理解成本

Agent 在多语言场景中仍然有统一工作心智。

### 4. 更好的长期扩展性

starter 不会因为语言或端增加而立刻失控。

---

## 8. 不达标时会是什么样子

如果这项能力没有做对，典型症状会是：

1. Starter 一多语言就乱
2. CLI 语义开始分裂
3. pack 安装规则开始因语言而变化
4. Agent 需要重新学习每个项目
5. 用户被迫理解技术实现细节

一旦出现这些问题，就说明“跨语言能力组合”还只是概念，不算真正交付成功。

---

## 9. 最终结论

Rendo 跨语言能力组合真正做成的标准，不是：

- 支持了多少语言

而是：

- 用户不需要为语言组合付出额外认知成本
- 强 Agent 仍然可以通过统一契约和统一 CLI 自然工作
- starter 仍然是清晰、可维护、可升级的统一工程

一句话结论：

**Rendo 跨语言能力组合“做成了”，意味着用户看到的是一个能力自然组合的 starter，Agent 看到的是一个统一语义的工程，而不是多个语言体系拼在一起的碎片。**
