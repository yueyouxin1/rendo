# Rendo 跨语言能力组合与统一 CLI 语义原则

- 文档版本：v1.1
- 日期：2026-04-04
- 文档性质：架构原则文档
- 目标：说明 Rendo 在多语言生态下应该做什么、为什么这样做、以及如何才能做对
- 核心结论：**Rendo 不应被实现为“每种语言各自一套平行体系”，而应被实现为“统一契约 + 统一 CLI 语义 + 多语言能力组合系统”。**

---

## 1. 问题定义

随着新定位继续收敛，一个关键问题越来越明确：

- Rendo 已经不再局限于 Web、`Next.js` 或单一语言生态
- 很多真实 AI 产品天然就是多语言组合

例如：

- 某些部分更适合用 `TypeScript`
- 某些部分更适合用 `Python`
- 某些端需要 Web 壳
- 某些端需要小程序壳
- 某些能力是 headless 的

因此，Rendo 面临的真实问题不是：

- 如何让一种语言统治全部 starter

而是：

- **如何在多语言、多形态、多能力组合的情况下，依然保持清晰、一致、可被强 Agent 理解。**

---

## 2. 核心判断

### 2.1 不该做什么

Rendo 不应做成：

- 每种语言一套独立 starter 世界
- 每种语言一套独立 CLI 语义
- 每种语言一套独立 pack 生态
- 每种语言一套独立平台规则

这种做法的问题是：

- 学习成本高
- 维护成本高
- Agent 理解成本高
- 整体体系会快速分裂

### 2.2 应该做什么

Rendo 应该把：

- 契约
- CLI 语义
- 模板描述方式
- pack 扩展方式
- runtime mode 语义

统一下来。

而把：

- 具体语言
- 具体框架
- 具体目录落位
- 具体实现方式

留给不同 starter 和 pack 在各自场景中合理实现。

一句话：

**统一的是规则，不是实现。**

---

## 3. 什么叫“跨语言能力组合”

Rendo 的跨语言支持，不应被理解成：

- 一个 starter 只能对应一种语言
- 或者每种语言必须各自完整封装一套体系

更合理的理解是：

- 一个 starter 可以由多种语言与多种执行单元组成
- 一个能力包也可以作用于不同语言实现
- 只要它们遵守统一契约和统一 CLI 语义，就应被视为同一体系的一部分

所以，“跨语言能力组合”本质上是在回答：

- **不同语言如何在一个 starter 体系里自然共存**

而不是回答：

- “哪种语言应该是默认中心”

---

## 4. 为什么“统一 CLI 语义”是关键

强 Agent 最不适合面对的情况是：

- 每种语言一套命令
- 每种 starter 一套行为
- 每种 pack 一套安装方式

它最适合面对的是：

- 相同命令
- 相同语义
- 相同可见性
- 相同失败方式

这意味着：

- `rendo init`
- `rendo create`
- `rendo inspect`
- `rendo add`
- `rendo pull`
- `rendo upgrade`
- `rendo doctor`

这些命令的语义必须跨语言保持一致。

语言不同可以影响：

- 具体生成内容
- 具体安装动作
- 具体依赖管理

但不应影响：

- 用户和 Agent 对命令的理解方式

一句话：

**CLI 是统一能力组合体验的最小控制面。**

---

## 5. 哪些东西必须统一

在跨语言体系中，以下部分必须统一：

## 5.1 Starter 身份语义

必须统一：

- 什么是 `Core Starter`
- 什么是 `Domain Starter`
- 一个 starter 如何声明自己的类型、能力和默认依赖

## 5.2 Pack 语义

必须统一：

- 什么是 pack
- pack 如何声明类型
- pack 如何声明运行模式
- pack 如何声明安装影响面

## 5.3 Runtime mode 语义

必须统一：

- `source`
- `managed`
- `hybrid`

的定义和边界。

## 5.4 CLI 语义

必须统一：

- `init` 是初始化底座
- `create` 是创建 starter
- `inspect` 是查看 starter 或 pack 说明
- `add` 是安装能力
- `doctor` 是诊断环境和依赖

## 5.5 配置与依赖表达方式

必须统一：

- 配置入口的概念
- provider 的声明方式
- 必要环境变量的表达方式
- install / upgrade / remove 的基本逻辑

---

## 6. 哪些东西不应该写死

为了避免把实现思维限制死，以下部分不应在原则层写死：

1. 固定目录名称
2. 固定一门语言做 UI
3. 固定一门语言做服务
4. 固定某种“app/service”分法
5. 固定某种单仓目录长相

原因很简单：

- 这些都属于实现策略
- 会随着 starter 场景和语言生态变化
- 过早写死会限制 Codex 和后续工程选择

原则文档应回答的是：

- 必须统一什么
- 必须保持什么边界
- 怎样做才不会混乱

而不是：

- 代码具体长什么样

---

## 7. 怎样做才算“做对”

如果 Rendo 要把跨语言能力组合做对，至少应满足以下条件：

## 条件 1：Agent 看到不同语言 starter 时，不需要重新学习一套世界

它应该仍然能通过：

- manifest
- CLI
- docs
- install plan

快速理解这个 starter。

## 条件 2：不同语言实现仍然能被纳入同一套模板与 pack 体系

无论某个能力是：

- `TypeScript`
- `Python`
- 其他生态

都必须可以被 starter 和 pack 体系统一描述。

## 条件 3：组合能力不应破坏 starter 的可理解性

即使一个 starter 是多语言的，也必须：

- 结构清晰
- 依赖明确
- 安装路径明确

而不能让强 Agent 陷入混乱。

## 条件 4：CLI 必须优先于底层实现差异

不同语言实现可以不同，  
但用户和 Agent 的操作方式不应该跟着语言大幅变化。

## 条件 5：契约必须先于实现

先定义：

- starter contract
- pack contract
- runtime mode
- CLI semantics

再去做多语言 starter。

---

## 8. 对 `Core Starter` 的影响

这个原则进一步说明：

`Core Starter` 的“语言无关”应理解为：

- 它定义统一工程语言和扩展规则

而不是：

- 它自己要实现所有语言版本

所以：

- `Core Starter` 应足够抽象
- `Domain Starter` 才去绑定具体实现形态

这让多语言支持成为：

- 自然扩展能力

而不是：

- 体系分裂点

---

## 9. 最终结论

Rendo 在跨语言场景下最重要的不是：

- 支持多少语言

而是：

- 是否能在多语言现实下依然保持统一认知、统一 CLI、统一 manifest 和统一扩展机制

一句话结论：

**Rendo 真正要做的不是“多语言 starter 集合”，而是“在统一契约和统一 CLI 语义之上的多语言能力组合系统”。**
