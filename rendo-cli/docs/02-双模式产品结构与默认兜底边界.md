# 双模式产品结构与默认兜底边界

## 核心问题

源码 starter 路线虽然更自然，但对完全不懂工程的小白用户来说，仍然会遇到：

- AI 凭证
- OAuth 凭证
- 渠道开放平台授权
- 第三方 API 凭证

这些复杂度不是代码问题，而是配置与所有权问题。

## 核心结论

Rendo 应同时支持两种消费模式：

### 1. Managed Mode

面向：

- 小白用户
- 不想处理技术细节的用户

特点：

- 默认模型
- 默认 Auth / Billing / KB / Agent / Workflow
- 平台兜底大部分技术性基础配置

### 2. Source Mode

面向：

- 强 Agent 用户
- 高级用户
- 工程团队

特点：

- 拉源码
- 可 eject
- 可替换默认能力
- Agent 可继续深度开发

## 哪些配置无法由平台完全兜底

以下通常仍然需要用户或业务主体自己授权：

- 飞书开放平台
- 微信开放平台
- 钉钉开放平台
- 某些企业 OAuth / 支付主体

平台可以做的是：

- 引导
- 校验
- deep link
- step-by-step 向导

## 哪些配置应由平台默认兜底

- AI 模型默认 provider
- 搜索与检索默认能力
- 默认 Agent / Workflow
- 默认 DB / cache
- 常见工具 API 的官方 pack

## 最终原则

**用户只处理“必须由业务主体亲自授权”的配置，其余技术性基础能力尽量由 Rendo 默认兜底。**
